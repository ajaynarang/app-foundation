# Running Tests

How to run each test suite. All `:local` variants inject secrets from Doppler (project `sally-backend`, config `dev`) so you don't maintain local `.env` files.

## Quick reference

| Command | Runs | Notes |
|---|---|---|
| `pnpm test` | Turborepo `test` task — backend Jest. Web is a no-op (Jest misconfigured). | Required: backend infra running for some specs |
| `pnpm backend:test` | Backend Jest only | Same as above, narrower |
| `pnpm test:qa` | Full Playwright suite | **You supply env vars** — `TENANT_ID`, `API_BASE_URL`, `WEB_BASE_URL` |
| `pnpm test:qa:local` | Full Playwright suite with Doppler injection | Just supply `TENANT_ID` |
| `pnpm test:smoke:local` | Smoke (`@smoke` tag, `--project=api`) | ~30s |
| `pnpm test:rbac:local` | RBAC matrix (`@rbac` tag, `--project=api`) | ~2min |
| `pnpm test:api:local` | API workflows + contracts (`tests/api`, `--project=api`) | ~3min |
| `pnpm test:browser:local` | Browser (`--project=browser`) | ~2min |
| `pnpm qa:list-tenants` | List valid tenant IDs for the target | Run this first to find a `TENANT_ID` |
| `pnpm qa:enable-features` | Enable QA-only feature flags on a tenant | Idempotent |
| `pnpm qa:report` | Generate the confidence matrix HTML report | After a test run |
| `pnpm tenant:reset` | Reset a tenant — wipe loads/drivers/financials in FK-safe order | Use with care |

## Before your first run

```bash
doppler login                              # one-time
cd apps/backend && doppler setup           # one-time per app, picks sally-backend/dev
cd ../web && doppler setup                 # sally-web/dev
cd ../console && doppler setup             # sally-console/dev
```

Make sure the dev stack is up (or you're pointing at staging):

```bash
docker-compose up -d                       # postgres + redis + inngest
pnpm doppler:backend                       # NestJS on :8001
```

Then:

```bash
pnpm qa:list-tenants
# Outputs: 1) TENANT_ID=abc-123, 2) TENANT_ID=def-456, ...
# Pick one.
```

## Local runs (Doppler-injected, no env files)

```bash
TENANT_ID=<id> pnpm test:smoke:local       # smoke only
TENANT_ID=<id> pnpm test:rbac:local        # role × endpoint matrix
TENANT_ID=<id> pnpm test:api:local         # workflow + contract tests
TENANT_ID=<id> pnpm test:browser:local     # Playwright browser
TENANT_ID=<id> pnpm test:qa:local          # all of the above
```

Backend Jest unit tests don't need a tenant:

```bash
pnpm backend:test                          # all backend specs
cd apps/backend && pnpm test -- --testPathPattern desk    # filter
cd apps/backend && pnpm test -- --watch                   # watch mode
cd apps/backend && pnpm test:cov                          # with coverage
```

## Manual env vars (bring-your-own)

If you're running against an environment that isn't covered by Doppler `sally-backend/dev` — for example, an ephemeral preview deployment:

```bash
TENANT_ID=<id> \
API_BASE_URL=https://preview-foo.sally.appshore.in/api/v1 \
WEB_BASE_URL=https://preview-foo.sally.appshore.in \
DEV_AUTH_SECRET=<secret> \
pnpm test:qa
```

The `DEV_AUTH_SECRET` is needed for the `DevAuthGuard`-backed role impersonation that the suite uses to test as different roles without real logins. Get it from the matching Doppler config.

## Running just one spec

```bash
# Playwright — Jest-style path filter
pnpm --filter @sally/qa exec playwright test tests/api/fleet/loads.spec.ts

# With grep
pnpm --filter @sally/qa exec playwright test --grep "dispatches load"

# Backend Jest
cd apps/backend && pnpm test -- tests/path/to/spec.test.ts
```

## Quality Gate from GitHub Actions

The `Quality Gate` workflow runs the same suites against a target environment. Always manual (`workflow_dispatch`).

1. Open the repo's Actions tab → **Quality Gate** → Run workflow.
2. Inputs:
    - `target` — API URL (defaults to `https://sally-api-staging.appshore.in/api/v1`).
    - `web_target` — frontend URL (defaults to staging).
    - `tenant` — required, the tenant to test against.
    - `suite` — `all` / `unit` / `smoke` / `rbac` / `api` / `browser`.
3. Watch the run, download artifacts when done. Reports land under `tests/reports/` in the artifact.

The workflow itself runs on Ubuntu with Node 22 (the unit-tests job pins 22; the rest of the runners use what they need).

## Reports

After a local QA run:

```bash
pnpm qa:report
# Generates tests/reports/confidence-matrix.html
```

After a CI run, the report is uploaded as an artifact and (for runs that land on `develop` and trigger a docs deploy) copied into the published docs at:

- `qa-report.html` at the doc site root — the **latest** run.
- `qa/reports/` — the history of runs, ordered newest-first.

These artifacts are generated by the `docs.yml` workflow's "Copy QA reports into site" step.

## Common failure modes

| Symptom | Likely cause | Fix |
|---|---|---|
| `TENANT_ID is required` | No env var set | `pnpm qa:list-tenants` to find one, then set `TENANT_ID` |
| All RBAC tests fail with 401 | `DEV_AUTH_SECRET` mismatched between client and server | Refresh Doppler and re-run |
| Browser tests can't reach the web | `WEB_BASE_URL` points somewhere wrong | Check the env you're targeting |
| One spec hangs forever | Probably waiting on a DB connection that never opened | `docker-compose ps` to confirm Postgres + Redis are healthy |
| `pnpm test` runs nothing | Backend Jest config not finding specs (`testPathIgnorePatterns` skipped them) | Check the 4 documented skips in [Backend → Testing](../backend/testing.md) |
