# Database & Prisma

The Prisma schema at `apps/backend/prisma/schema.prisma` is the single source of truth for relational data. Migrations are generated by Prisma; they are **applied** by a wrapper script.

## The one hard rule

**Don't run `prisma migrate dev` locally.** It uses `db push` semantics for drift and will reset mastra-owned tables that Prisma doesn't know about — wiping agent memory and chat history. Always use the wrapper script:

```bash
./tools/db/migrate.sh --env local --migrate-only -y
```

The wrapper uses `prisma migrate deploy`, which is idempotent and drift-tolerant. There's no scenario where `prisma migrate dev` is the right call.

## Adding a schema change

1. **Edit `apps/backend/prisma/schema.prisma`.** Add the model / field / enum value.

2. **Generate a migration.** From `apps/backend/`:

    ```bash
    doppler run -- pnpm exec prisma migrate dev --create-only --name <descriptive_name>
    ```

    > `--create-only` makes Prisma write the SQL but NOT apply it. We never let Prisma's migrate flow touch the database directly. The wrapper script below applies it safely.

3. **Inspect the generated SQL** at `apps/backend/prisma/migrations/<timestamp>_<descriptive_name>/migration.sql`. Make sure it does what you expect. For destructive operations (drop column, change type), add a `BEGIN ... COMMIT` block manually if you want safety.

4. **Apply locally:**

    ```bash
    ./tools/db/migrate.sh --env local -y
    ```

    Full mode runs migrations, then seeds, then the Langfuse-style prompt seed. Add `--migrate-only` to skip seeds.

5. **Regenerate the client + shared-types mirror:**

    ```bash
    cd apps/backend && doppler run -- pnpm prisma:generate
    pnpm --filter shared-types build
    ```

    `prisma:generate` chains `prisma generate` with `tsx scripts/generate-shared-enums.ts`, so frontend / QA see new enum values immediately.

6. **Commit the migration directory** along with the schema change. They're version-controlled.

## The `tools/db/migrate.sh` flags

```
./tools/db/migrate.sh --env <local|staging> [OPTIONS]
```

| Flag | Effect |
|---|---|
| (no mode flag) | Full mode: migrate + seed + Langfuse seed |
| `--migrate-only` | Apply migrations only |
| `--seed-only` | Run seeds only (no migrations) |
| `--seed-langfuse` | Seed Langfuse prompts only |
| `--no-langfuse` | Full mode but skip Langfuse |
| `--status` | Show migration & seed status (read-only) |
| `--dry-run` | Show pending changes, don't apply |
| `--yes`, `-y` | Skip confirmation prompts |
| `--container NAME` | Override the staging ECS jump-host container (default `worker`) |

For `--env staging`, the script opens an SSM tunnel to the staging RDS instance, applies, then tears down the tunnel. Staging credentials are read from `apps/backend/.env.aws.staging` (gitignored). Never hardcode credentials.

## Seeds

Three seed scripts under `apps/backend/scripts/` / `apps/backend/prisma/`:

| Script | What it does | Run via |
|---|---|---|
| `apps/backend/prisma/seeds/index.ts` | Base seed — feature flags, default plans, default fuel-card types | `pnpm setup:base` |
| `apps/backend/scripts/demo/index.ts` | Demo tenant + loads, drivers, vehicles, customers | `pnpm setup:demo` (add `--reset` to wipe first) |
| `apps/backend/scripts/seed-langfuse-prompts.ts` | LLM prompts seeded into the `prompting/` domain tables | `pnpm seed:langfuse` |

All three run automatically as part of `./tools/db/migrate.sh --env local` (full mode).

## Common Prisma patterns

### Tenant scoping

Every domain query must filter by tenant. The convention is `tenant_db_id` (numeric internal ID) or `tenant_id` (string public ID), depending on the model:

```ts
const loads = await this.prisma.load.findMany({
  where: { tenant_db_id: tenantDbId, status: 'DISPATCHED' },
  orderBy: { created_at: 'desc' },
});
```

### camelCase boundary

Prisma blocks are snake_case. Returned objects are camelCase. Map at the service:

```ts
const load = await this.prisma.load.findUnique({ where: { id } });
return {
  id: load.id,
  customerName: load.customer_name,
  pickupDate: load.pickup_date,     // string in YYYY-MM-DD — see Data Model
  createdAt: load.created_at.toISOString(),
};
```

### Includes vs select

Prefer `select` for performance and explicit shape — `include` returns every column of the related model. For nested includes, name only what you need.

### Transactions

For multi-write operations, use `this.prisma.$transaction(...)`:

```ts
await this.prisma.$transaction([
  this.prisma.invoice.create({ data: invoiceData }),
  this.prisma.invoiceLine.createMany({ data: lineData }),
  this.prisma.load.update({
    where: { id: loadId },
    data: { invoice_id: invoiceData.id },
  }),
]);
```

Or the callback form when later writes depend on earlier results:

```ts
await this.prisma.$transaction(async (tx) => {
  const invoice = await tx.invoice.create({ data: invoiceData });
  await tx.invoiceLine.createMany({ data: lineData.map((l) => ({ ...l, invoice_id: invoice.id })) });
  return invoice;
});
```

### pgvector

For embeddings (RAG over uploaded docs), the schema uses raw `Bytes`/`Unsupported` columns wrapped by helper services in `apps/backend/src/domains/ai/knowledge-base/`. Don't write to vector columns directly — go through the helper.

## Status & reset

```bash
./tools/db/migrate.sh --env local --status     # what's applied
./tools/db/migrate.sh --env local --dry-run    # what would apply

# Nuclear option — wipe and re-seed local DB
cd apps/backend && doppler run -- pnpm db:reset
```

`db:reset` runs `prisma migrate reset --force` and is the one legitimate use of Prisma's reset flow (intentional wipe).

## Production migrations

Production migrations are part of the `Deploy All` workflow. The backend container runs `prisma migrate deploy` at startup against the production DATABASE_URL (read from Doppler `sally-backend/prd`). The Terraform config in `infra/terraform/` defines the task definition and IAM.

For staging, you can pre-flight a migration without deploy by running `./tools/db/migrate.sh --env staging --dry-run`. The script opens the SSM tunnel automatically.
